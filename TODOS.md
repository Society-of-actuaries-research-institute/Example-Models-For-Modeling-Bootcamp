## TODO-1: Performance benchmark job for 100k×10k scale

**What:** A pytest mark `@pytest.mark.slow` (or a separate GitHub Actions nightly workflow)
that runs the full 100k×10k projection and asserts completion in under 10 minutes.

**Why:** The CI suite (coverage, mypy, pylint) never runs the scale target. A regression
in batch loop efficiency would pass all tests. This is the only way to mechanically verify
the most important non-functional requirement.

**Pros:** Protects the 10-minute SLA. Makes scale regression visible.

**Cons:** Too slow for the free GitHub Actions tier as a per-PR check. Realistically a
`@pytest.mark.slow` skip-by-default test or a nightly scheduled workflow.

**Context:** Start by adding `@pytest.mark.slow` in `pyproject.toml`'s pytest markers
config. The test: `assert run_time < 600` seconds, inputs from a synthetic 100k-policy
fixture generated by a helper function.

## TODO-2: `--output-dir` CLI flag for runner.py

**What:** A `--output-dir PATH` argparse flag so students can control where
`results_{timestamp}.xlsx` is written, rather than always using CWD.

**Why:** Students running from VS Code's default terminal directory vs the repo root
see the output file in an unexpected location. A two-line argparse addition eliminates
the confusion.

**Pros:** Better developer experience for students. No ambiguity about output location.


**Cons:** Minor scope addition. Current CWD behavior is acceptable for a teaching tool.

**Context:** Add to `runner.py`'s argument parser in Step 6. Default: `Path(".")`.
Pass the resolved output path to `ReportWriter`.

**Depends on:** Step 6 (runner.py) complete.
